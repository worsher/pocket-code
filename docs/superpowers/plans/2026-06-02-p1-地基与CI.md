> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 让整个仓库编译干净、现有测试全绿，并建立真正会运行的 CI 门禁（修复触发分支错配 + 覆盖全部包），为后续 P2–P5 提供可靠地基。

**Architecture:** 不引入新功能，只做四件事：① 修 3 处真 bug（daemon 过时测试、app fileTransfer 的 expo-fs 迁移、app notifications 权限 API）；② 删除残留空目录；③ 加根级 build/test/typecheck 脚本；④ 修 `ci.yml` 在 `master` 触发并跑全部包的构建与测试。app 的类型检查用 grep 过滤掉已知的 `expo-modules-core` 预存噪音，只对真错误失败。

**Tech Stack:** pnpm workspace、TypeScript（tsc）、vitest、expo-file-system 19（SDK 54）、expo-notifications、GitHub Actions。

## 背景事实（执行者必读）

- 仓库默认分支是 **`master`**，但 `.github/workflows/ci.yml` 只在 `main` 触发 → CI 从未真正运行，所以 daemon 的过时测试一直漏网。
- `pnpm test`（根脚本）只跑 server；daemon/relay/wire 测试未进 CI。
- app 跑 `tsc --noEmit` 有 **12 个错误**：其中 **4 个是真 bug**（本计划修复），**8 个是 `expo-modules-core` 找不到的预存噪音**（pnpm 传递依赖未提升到 app 的类型解析路径，非本项目引入，本计划用 grep 过滤而不强行解决）。
- expo-file-system 19 把旧 API 移到了 `expo-file-system/legacy` 子路径；旧 API（`cacheDirectory`/`createUploadTask`/`downloadAsync`/`FileSystemUploadType`）从该子路径继续可用。这是官方迁移路径，比改写为新 `File/Paths` API 风险低。
- daemon 配对码实现是正确的：8 位、字符集 `0123456789ABCDEFGHJKLMNPQRSTUVWXYZ`（34 字符，排除 I/O），见 `packages/daemon/src/pairing.ts:51-52`。错的是测试断言。

export async function requestNotificationPermissions(): Promise<boolean> {

- [ ] **Step 1: 创建过滤已知噪音的 app 类型检查脚本**

创建 `scripts/typecheck-app.sh`，内容：

```bash
#!/usr/bin/env bash
# 类型检查 app，忽略已知的 expo-modules-core 传递依赖解析噪音
# (pnpm 未把 expo-modules-core 提升到 app 的类型解析路径，非本项目引入)。
# 任何其它(真)类型错误都会导致退出码 1。
set -uo pipefail
cd "$(dirname "$0")/../packages/app"
OUT=$(npx tsc --noEmit 2>&1 | grep "error TS" | grep -v "expo-modules-core" || true)
if [ -n "$OUT" ]; then
  echo "App 存在真实类型错误(已排除 expo-modules-core 噪音):"
  echo "$OUT"
  exit 1
fi
echo "App typecheck OK (仅剩已知 expo-modules-core 噪音)。"
```

赋予可执行权限：

Run: `chmod +x scripts/typecheck-app.sh`
