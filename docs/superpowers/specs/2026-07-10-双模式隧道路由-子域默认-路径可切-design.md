# 双模式隧道路由(子域默认 + 路径前缀可切)设计

> **状态**:设计已确认(2026-07-10),待写实现计划。
> **前置**:relay 拆分(protocol-core + tunnel-client + TUNNEL_TOKEN/DISCOVERY 加固)已完成并合入 master。

## 背景与问题

反向隧道预览当前只有**路径前缀**一种寻址:`/t/<machineId>/<port>/<path>`,配 `pc_tunnel=<machineId>:<port>` cookie 兜底绝对路径子资源(P5 设计)。

真机暴露的问题:访问 `/t/m_bda488cfdee21ccc/3000`(Payload CMS)后,浏览器地址栏变成裸 `/admin`,页面虽能渲染,但:

1. **URL 掉出 `/t/` 前缀** —— 整个会话可达性依赖 `pc_tunnel` cookie 存在。
2. **链接不可分享** —— 把 `https://aigc.zj.cn/admin` 发给别人,对方无 cookie → 404 或被自己的 cookie 路由到错误机器。
3. **多机器/多端口并用会串** —— cookie 单值 `machineId:port`,后种覆盖前者,裸绝对路径全被路由到最后那台机器。

根因:`pc_tunnel` cookie 在**裸 origin 路径**(`/admin`、`/_next/*`)上路由,不带"当前属于哪个 `/t/` 前缀"的信息。Payload/Next 这类应用假设自己独占站点根(发 origin 绝对路径的 Location 与资源 URL),路径前缀方案天然容纳不了它们的整页导航。

取证结论(本会话逐跳验证):
- 直连 `localhost:3000/` → **307 Location: /admin**。
- 经隧道 `/t/.../3000` → **200**(无 Location):daemon 的 `proxyToLocalhost` 用 undici `fetch`,默认 `redirect:"follow"`,在开发机本地就把 307 跟随掉了。所以那次 URL 变化是 **Payload admin 的客户端 SPA 路由**(`pushState`),不是服务端 307。
- 带 `pc_tunnel` cookie 请求 `/admin` → 200:证实 cookie 兜底路由在工作。

## 目标

- **默认提供完整解法**:子域模式(`<machineId>-<port>.tunnel.<domain>`),应用绝对路径天然正确、链接完全可分享、SPA `pushState` 也不掉前缀。
- **保留低成本回退**:路径前缀模式(`/t/<id>/<port>/...`),用于裸 IP / 无泛证书的部署;在此模式下新增 `Location` 补前缀,让**服务端重定向类**(HTTP 3xx)不掉前缀。(多机不串靠子域模式,非本回退目标 —— 见"非目标"。)
- 两模式由环境变量切换,**默认 `subdomain`**。
- 收编既有偏差:daemon fetch 吞 3xx。

## 非目标(诚实的能力边界)

以下均为**路径模式**的已知限制;**子域模式全部不受影响**(根即根)。要规避这些限制,用子域模式(默认)。

- **SPA 纯客户端 `pushState('/admin')` 不掉前缀做不到** —— relay 只能改它看得见的(HTTP `Location` 头)。SPA 加载后 JS 自己改地址栏那一跳 relay 拦不到。
- **多机器 / 多端口并用会串** —— `pc_tunnel` cookie 为兜底裸绝对路径子资源必须保持 `Path=/`(全站单值),后开的隧道 cookie 覆盖前者。收窄 Path 能隔离多机,但会让 HTML 内裸绝对路径子资源(`/_next/*`)带不上 cookie → 404,弄坏单机预览,故不收窄。**多机不串的完整解是子域模式**。
- **不重写响应 body**、不注入脚本(`<base href>` / History API 补丁)—— 脆弱且侵入应用。
- **不改写 Set-Cookie**(见 redirectRewrite):只补 `Location`。
- 不做别名注册表 / 短别名(YAGNI)。子域直接用 machineId。
- 不为路径模式做 HTML 内绝对路径(`/_next/*`)重写 —— 仍靠 `pc_tunnel` cookie(`Path=/`)兜底子资源(子资源不改地址栏,不影响可分享)。

## Global Constraints

- **默认模式 = `subdomain`**;`TUNNEL_MODE=path` 显式切换到路径前缀模式。判定沿用现有 `isDiscoveryEnabled` 风格(默认值 + 仅特定字符串切换,大小写/空格不敏感)。
- **machineId 改为纯 16 位 hex**(去掉 `m_` 前缀),使其 DNS-safe。两处生成点必须同时改:`packages/daemon/src/index.ts` 与 `packages/tunnel-client/src/identity.ts`。
- 子域模式下 `TUNNEL_BASE_DOMAIN` **必需**;缺失则启动**快速失败**(抛错带指引),与 `requireRelaySecret` 同款硬失败,不静默降级。
- **鉴权语义两模式一致**:`TUNNEL_TOKEN` 判定、404 uniform、`pc_token` 剥离、`TUNNEL_COOKIE_SECURE` 开关全部与寻址模式正交、两模式共用。
- **路径模式补前缀只作用于 HTTP `Location` 头**;**不改写任何 Set-Cookie**(包括 relay 自己的 `pc_tunnel`/`pc_tunnel_token` 与应用自身的 cookie)。`pc_tunnel` 保持 `Path=/`(保底 HTML 内裸绝对路径子资源的兜底路由,见"路径模式已知限制")。
- HTTP 与 WS-upgrade 两个入口必须共享同一寻址语义(消除现有两份漂移的正则/cookie 逻辑)。
- TDD:每个纯函数模块 RED→GREEN。测试不得断言空。
- 许可证 MIT(沿用 relay 包)。

## 架构

### 模式对照

| 模式 | `TUNNEL_MODE` | URL 形态 | 寻址来源 | cookie 兜底 | 补前缀 | 适用 |
|---|---|---|---|---|---|---|
| 子域(默认) | `subdomain` | `<machineId>-<port>.tunnel.<domain>/<path>` | **Host 头** | 不需要(根即根) | 不需要 | 有泛域名+泛证书 |
| 路径前缀 | `path` | `/t/<machineId>/<port>/<path>` | **URL path** | `pc_tunnel` cookie(`Path=/`) | 仅 `Location` 补 `/t/<id>/<port>` | 裸 IP / 无泛证书 |

**核心洞察**:两模式只在 relay 入口"解析 machineId+port"这一层分叉,下游(tunnelHub、daemon 转发、鉴权判定)完全共享。

### 数据流对照

```
浏览器请求
  │
  ├─[subdomain] Host: bda488cfdee21ccc-3000.tunnel.aigc.zj.cn, path=/admin
  │     → resolveTunnelTarget 解析出 (bda488cfdee21ccc, 3000, "/admin")
  │     → 转发 daemon → localhost:3000/admin
  │     → 3xx Location:/admin 原样透传(根即根,浏览器跳 /admin 仍在本子域)  ✅ 可分享
  │
  └─[path]  GET /t/bda488cfdee21ccc/3000/  (+ pc_tunnel cookie 兜底裸绝对路径子资源)
        → resolveTunnelTarget 解析出 (bda488cfdee21ccc, 3000, "/")
        → 转发 daemon → localhost:3000/
        → 3xx Location:/admin → redirectRewrite → Location:/t/bda488cfdee21ccc/3000/admin  ✅ 服务端重定向不掉前缀
        →(Set-Cookie 不改写;pc_tunnel 保持 Path=/ 兜底子资源;多机并用会串 = 已知限制,用子域模式)
```

## 文件结构

### 新增 `packages/relay/src/tunnelAddress.ts` —— 唯一寻址决策点

把"如何识别一个请求属于哪条隧道"从两个 router 抽出成纯函数。

```ts
export type TunnelMode = "subdomain" | "path";

export interface TunnelTarget {
  machineId: string;
  port: number;
  /** 转发给 daemon 的 path(不含 pc_token,含其余 query)。子域=原样 pathname;路径=剥前缀后的 rest。 */
  forwardPath: string;
}

export type ResolveResult =
  | { kind: "tunnel"; target: TunnelTarget }
  | { kind: "control" }   // 非隧道请求(控制通道 /relay,或子域模式下的主站 Host)
  | { kind: "none" };     // 既非隧道也非控制(HTTP 该 404)

/**
 * @param mode        隧道寻址模式
 * @param host        req.headers.host(子域模式用)
 * @param pathname    url.pathname
 * @param cookieHeader req.headers.cookie(路径模式 pc_tunnel 兜底用)
 * @param baseDomain  TUNNEL_BASE_DOMAIN(子域模式必需,如 "tunnel.aigc.zj.cn")
 */
export function resolveTunnelTarget(
  mode: TunnelMode,
  host: string | undefined,
  pathname: string,
  cookieHeader: string | undefined,
  baseDomain: string | null
): ResolveResult;
```

行为:
- `mode="subdomain"`:
  - Host 匹配 `<machineId>-<port>.<baseDomain>`(machineId = hex,port = 数字)→ `{kind:"tunnel", forwardPath: pathname}`。
  - Host = baseDomain 本身、或主站域、或裸 IP、或不匹配 → `{kind:"control"}`(交回控制通道判断 `/relay` 等)。
  - 忽略 `/t/` 前缀与 `pc_tunnel` cookie。
- `mode="path"`:
  - `/relay` 或 `/relay/` → `{kind:"control"}`。
  - `pathname` 匹配 `/^\/t\/([^/]+)\/(\d+)(\/.*)?$/` → tunnel,`forwardPath = (rest||"/")`。
  - 否则 `pc_tunnel` cookie 命中 → tunnel,`forwardPath = pathname`。
  - 都不命中 → `{kind:"none"}`。
  - 忽略 Host 子域。
- **不在此函数做**:`pc_token` 剥离、query 拼接、鉴权 —— 这些由调用方(router)在拿到 target 后统一做,保持本函数纯粹。

`httpRouter.ts` 与 `upgradeRouter.ts` 都改为调用它,删除各自内联的正则/cookie 逻辑。

### 新增 `packages/relay/src/redirectRewrite.ts` —— 仅路径模式

```ts
/** 路径模式:把隧道回来的响应 Location 头里的站内绝对路径补上 /t/<id>/<port> 前缀。子域模式不调用。 */
export function rewriteRedirectHeaders(
  headers: Record<string, string | string[]>,
  machineId: string,
  port: number
): Record<string, string | string[]>;
```

规则(**只改 `Location`,不碰任何 Set-Cookie**):
- `Location`:仅当**站内绝对路径**(`/` 开头、不以 `//` 开头、不是绝对 URL `scheme://`)→ 前缀 `/t/<machineId>/<port>` + 原值。相对路径、`//host`、`https://...` 原样不动。
- **不改写 Set-Cookie**:`pc_tunnel` 需保持 `Path=/` 以兜底 HTML 内裸绝对路径子资源(见"路径模式已知限制");应用自身 cookie 也不动(浏览器按响应 URL 的路径归属处理,不影响子资源可达)。
- 纯函数,只可能改动 `Location` 一个键,其余原样。

> **为什么不补 cookie 的 Path**:收窄 `pc_tunnel` 的 Path 到 `/t/<id>/<port>/` 虽能隔离多机,但会让 HTML 里写死的裸绝对路径子资源(`/_next/*`)请求带不上 cookie → 404,反而弄坏单机预览。取舍见"路径模式已知限制"。多机不串的完整解是子域模式。

### 修改 `packages/tunnel-client/src/tunnel.ts`

`proxyToLocalhost` 的 `init` 加 `redirect: "manual"`:

```ts
const init: RequestInit = { method: req.method, headers: reqHeaders, redirect: "manual" };
```

效果:3xx+Location 不再被本地 fetch 跟随,原样成帧回传。响应头透传(剥 content-encoding/length)不变。这是"补前缀"与"子域下重定向正确"的共同前置。

## 鉴权一致性(逐条)

- 判定不变:`tunnelToken===null || verifyTunnelToken(queryToken) || verifyTunnelToken(cookieToken)`,与寻址模式正交。
- `pc_tunnel_token` cookie 作用域:
  - 子域模式:天然按 Host 隔离,每条隧道独立鉴权态(白送的更强隔离)。
  - 路径模式:`Path=/`(全站)不变;补前缀**不动** `pc_tunnel_token` 的 Path。
- 404 uniform:两模式鉴权失败一律 404。子域模式区分"Host 不匹配泛域"(→ control,非隧道)与"Host 匹配但 token 错"(→ 404,是隧道但无权限)。
- `pc_token` 剥离:两模式都从转发 path 剥掉,`resolveTunnelTarget` 之后统一做。
- `TUNNEL_COOKIE_SECURE`:不变,两模式共用。

## 错误处理

- 子域模式缺 `TUNNEL_BASE_DOMAIN` → 启动快速失败。
- Host 解析边界(端口非数字、machineId 空、段数不对)→ `{kind:"control"}` 或 `{kind:"none"}`,不 500。
- daemon `redirect:manual` 后 3xx body 通常为空 → relay 成帧逻辑需容忍空 body(测试覆盖)。
- 相对 `Location`(子域模式)由浏览器基于子域正确解析,不改写。

## 部署:子域模式证书(Cloudflare DNS-01 泛证书,推荐)

要覆盖 `<machineId>-<port>.tunnel.aigc.zj.cn`(即 `aigc.zj.cn` 的**二级**子域通配 `*.tunnel.aigc.zj.cn`)。关键事实:

- Cloudflare **免费 Universal SSL 不覆盖二级子域通配**(只覆盖根 + 第一级 `*.aigc.zj.cn`)。
- **推荐路径**:Cloudflare 只当 DNS 供应商,证书由 **Let's Encrypt DNS-01** 签(免费,支持任意层级通配),TLS 在 VPS 的 nginx 终止 —— 延续现有 `wss://aigc.zj.cn/relay` 的自管 nginx 方案。
  1. Cloudflare 建 API Token,权限限 `Zone.DNS:Edit`(zone `zj.cn`)。
  2. VPS 装 `certbot` + `python3-certbot-dns-cloudflare`,用该 token 跑 DNS-01 签 `*.tunnel.aigc.zj.cn`(+ `tunnel.aigc.zj.cn`)。
  3. nginx 加 `server { server_name *.tunnel.aigc.zj.cn; ... }` 反代到 relay,证书指向泛证书;certbot 定时续签。
- 备选:Cloudflare 付费 ACM(~$10/月)可签二级通配并在边缘终止(需开橙云代理)。
- 排除:免费 Universal SSL 直接代理(二级子域证书报错)。
- 泛 DNS 记录 `*.tunnel.aigc.zj.cn` → relay 所在 VPS IP(A/AAAA)。
