# 真机验收指南（P6a 安全加固 / P6b 协议归一化 / P7 HMR 隧道）

> 适用版本：`refactor/architecture-redesign` 分支 `26db4ec` 及之后。
> 初次部署（systemd/pm2/nginx 完整配置）见 `deployment-relay-daemon.md`；本文只讲**更新 + 验收**。

## 0. 重要变更须知（更新前必读）

1. **RELAY_SECRET 从可选改必填**（P6a）：relay 与 daemon 启动时未设置会**直接报错退出**。
   两端必须配同一个值：`openssl rand -hex 32` 生成一次，两边 export。
2. **协议不兼容旧版 App**（P6b 直切）：server/daemon 事件字段已归一化（`toolName→name` 等），
   **旧 App 配新 daemon 会显示异常**——App 必须与 relay/daemon 同批更新。
3. **nginx 用户**：`/t/` location 需要补 WS upgrade 头（P7 HMR），见 §1.3。

## 1. 更新步骤

### 1.1 开发机（Daemon）

```bash
cd pocket-code
git fetch && git checkout refactor/architecture-redesign && git pull
pnpm install
pnpm build                        # wire→server→relay→daemon 拓扑序构建

# 环境变量确认(新增必填项)。两种方式任选:
# 方式一:仓库根放 .env 文件(启动时的工作目录),内容一行一个 KEY=VALUE
echo "RELAY_SECRET=<与 relay 同一个值>" >> .env
# 方式二:export RELAY_SECRET='<值>'
# 其余(RELAY_URL / MACHINE_NAME / AI Key / CLAUDE_CLI_PATH)不变,也可一并写进 .env

pm2 restart pocket-daemon         # 或你的启动方式;裸跑: pnpm dev:daemon
```

启动成功标志：打印机器信息框 + 8 位配对码。
若打印 `[Daemon] 启动失败:RELAY_SECRET 未设置` → 按提示设置后重启（这是预期的 fail-fast）。

### 1.2 VPS（Relay）

```bash
cd /opt/pocket-code
git fetch && git checkout refactor/architecture-redesign && git pull
pnpm install
pnpm --filter @pocket-code/wire build && pnpm --filter @pocket-code/relay build

echo "RELAY_SECRET=<同上>" >> .env  # 或写进 systemd/pm2 配置;.env 放启动工作目录
pm2 restart pocket-relay          # 或 systemctl restart pocket-relay
```

启动成功标志：`[Relay] Listening on ws://0.0.0.0:3200`。
未设 secret 会打印 `[Relay] 启动失败:RELAY_SECRET 未设置...` 并退出（预期）。

### 1.3 nginx（有 TLS 反代的部署）

`location /t/` 块补两行后 `nginx -s reload`（完整配置见部署文档 §2.4）：

```nginx
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
```

### 1.4 手机 App

```bash
pnpm --filter @pocket-code/app exec expo run:android   # 或你现有的打包方式
```

App 代码在 P6b/P7 中公开接口零变化，但**必须重新打包**（内部消费的事件协议已切换）。

## 2. 验收清单

### A组：P6a 安全加固（约 10 分钟）

| # | 步骤 | 预期 |
|---|---|---|
| A1 | daemon/relay 不设 RELAY_SECRET 启动 | 立即退出，错误信息含 `openssl rand -hex 32` 指引 |
| A2 | 两端设一致 secret 启动 → App 设置页配对（输入 8 位码） | 配对成功，daemon 打印 `✅ Device ... paired successfully!` |
| A3 | 两端 secret 不一致时重启 daemon | daemon 反复重连被拒，relay 日志 `Invalid authToken. RELAY_SECRET mismatch.` |
| A4 | 配对后正常对话一轮 | 流式回复正常（顺带验证转发链路） |
| A5 | 删掉 daemon 的设备授权（`~/.pocket-code` 下 devices 文件删除后重启 daemon），App 发消息 | App 弹"需要重新配对"提示且**停止自动重连**（不会刷屏 daemon 日志）；重新配对后恢复 |

### B组：P6b 协议归一化（约 20 分钟，三模式各一轮）

**cloud/relay 模式（配对开发机，走中继）：**

| # | 步骤 | 预期 |
|---|---|---|
| B1 | 发"读取 X 文件并总结"（触发工具调用） | 工具卡片显示调用参数**和结果**（本次修复点：旧版结果常配不上对）；思维链正常流式 |
| B2 | 发"修改某文件"（写文件） | Diff/Files 页看到变更；local 工作区模式下文件自动同步到手机本地副本 |
| B3 | 长回复中途点停止 | 立即停止，界面收敛不卡"生成中" |
| B4 | 关 wifi 发一条消息 → 恢复网络 | 消息显示 pending → 重连后**自动重发并得到回复**（本次修复点：旧版离线重放发的是非法消息） |
| B5 | 手机后台时让 agent 跑 runCommand | 收到"命令执行完成/失败"系统通知 |
| B6 | 会话历史：切换会话/新建会话/编辑重发 | 行为与更新前一致 |

**LAN 直连模式**：B1-B3 重复一遍（连 `ws://开发机IP:3100` 或你的直连配置）。

**geek+local 模式（端侧 loop）：**

| # | 步骤 | 预期 |
|---|---|---|
| B7 | 配好 API Key 后对话 + 触发本地文件工具 | 对话/工具卡片渲染与 cloud 模式观感一致（共用同一渲染逻辑） |
| B8 | 后台跑 runCommand（走 WS 回退） | 同样收到系统通知（本次修复点） |

### C组：P7 HMR 隧道（约 10 分钟）

前置：开发机上有一个 vite 项目，`npm run dev` 起在某端口（如 5173）。

| # | 步骤 | 预期 |
|---|---|---|
| C1 | relay 模式打开 Preview（隧道 URL `/t/<machineId>/5173/`） | 页面正常渲染（HTTP 隧道回归） |
| C2 | 开发机改一个组件文件保存 | 手机 WebView **不整页刷新即更新**（HMR 生效）。若整页刷新或无反应 → 见 §3 排障 |
| C3 | 杀掉 dev server 再启动 | 手机页面短暂断连后**自动恢复热更**（vite client 自动重连重建隧道） |
| C4 | App 对话/配对照常使用 | 控制通道无回归（本次动了底层 WS 挂载方式） |
| C5 | （可选）next 项目重复 C1-C2 | `/_next/webpack-hmr` 走 cookie 路由，同样热更 |

局域网直连预览不经隧道，HMR 本来就通，不在本清单内。

## 3. 排障：关键日志速查

| 现象 | 看哪里 | 日志特征 → 含义 |
|---|---|---|
| daemon 连不上 relay | relay 日志 | `Registration requires authToken` / `Invalid authToken` → secret 未设/不一致；`Registration timestamp expired` → 两机时钟差 >5 分钟 |
| App 一直重连 | App 日志 / daemon 日志 | daemon 打印 `Unauthorized: invalid or revoked device token` → 设备授权失效，App 应已弹重新配对提示 |
| HMR 无反应（整页刷新才更新） | relay 日志 | 无 `tunnel-ws-open` 相关输出 → WS upgrade 没到 relay：nginx 缺 `/t/` 的 Upgrade 头（§1.3），或 WebView 的 WS 握手没带 `pc_tunnel` cookie（先开 C1 的页面再看） |
| HMR 断流 | daemon 日志 | `WS tunnel ... error` → dev server 端口/路径问题；`Dropped ws-tunnel data ... (state=...)` → 本地连接未就绪时来了数据（偶发无害，频繁则报我） |
| 疑似伪造/串线 | relay 日志 | `Dropped forged ...` / `Dropped ws-tunnel frame ... from non-owner` → 身份绑定在工作（正常不应出现，出现说明有另一个 daemon 在用相同 machineId） |
| 预览页资源 404 | relay 日志 | 绝对路径子资源依赖 `pc_tunnel` cookie——必须先经 `/t/<machineId>/<port>/` 入口打开页面 |

## 4. 已知限制（验收时不算失败）

- gemini-cli 路径已归一化但**无真机验证**（无订阅），claude-code 是主验证路径。
- `list-machines` 对任意 relay 连接可见（机器名/在线状态）——已知后置项。
- 大文件重命名后同步会走"删+增"（全量传新文件），非轻量 rename——设计取舍。
- HMR 隧道单元/集成测试用合成 ws server，真实 dev server 行为以本清单 C 组为准。

## 5. 验收结果反馈

任何一条不符合预期：把**现象 + 对应端的日志片段**（relay/daemon 关键行即可）发回来。
全部通过后下一步：合并 `refactor/architecture-redesign` → `master` 并让 CI 生效。
